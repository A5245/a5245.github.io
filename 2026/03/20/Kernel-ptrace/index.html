Kernel ptrace

Kernel ptrace

本文的目标是：分析ptrace运行原理。下文中使用Linux Kernel 5.10.245为蓝本构建。

ptrace() 系统调用是 Linux 提供的一个调试进程的工具，ptrace() 系统调用非常强大，它提供非常多的调试方式让我们去调试某一个进程，下面是 ptrace() 系统调用的定义：

附加

通过PTRACE_ATTACH、PTRACE_SEIZE或PTRACE_TRACEME调用ptrace启用调试附加。

首先了解在task_struct中ptrace设置的状态，下面是task_struct中与ptrace有关的几个字段。

ptrace_attach在使用SEIZE模式下不会停止附加进程，设置task的ptrace字段标记当前被ptraced。设置task状态并唤醒等待状态转换。

ptrace_link将tracee的parent替换为ptracer，后续的通知parent都转到ptracer上。将tracee加入ptraced链表中。

信号劫持

接下来tracee来到signal.c的信号处理，这个时候已经收到ptrace附加后的SIGSTOP信号等待处理。

进入ptrace_signal等待信号处理。唤醒后的exit_code作为替换信号值，为0表示抑制该信号不传递给task对应的信号handler。同时也可以更改信号值。

进入ptrace_stop时会将中将task状态设置为TASK_TRACED，exit_code作为临时变量存储信号值，如果处于ptrace状态通知ptracer状态变化。并将自身调度出CPU等待唤醒。

使用SIGCHLD信号通知ptracer变动。

ptracer在收到SIGCHLD后使用waitpid或其它wait变体，获取停止事件。waitpid最终进入exit.c的do_wait函数中。

do_wait中的重点在遍历当前ptraced的task

进入wait_consider_task，task当前exit_state不为EXIT_DEAD或EXIT_ZOMBIE。EXIT_DEAD和EXIT_ZOMBIE是退出时详细讲解的。

在wait_task_stopped中获取exit_code编码(exit_code << 8) | 0x7f后返回给用户态上。这里对应WIFSTOPPED检查处理，告诉waiter当前task已经stopped

ptracer使用PTRACE_GETSIGINFO获取last_siginfo的信号数据。

ptracer处理完成后可使用PTRACE_SINGLESTEP或PTRACE_CONT唤醒调试task恢复运行。

退出

这里退出需要区分两种情况：

  1. ptracer主动使用detach分离调试
  2. ptracer或tracee的task结束运行退出

首先分析第一种情况：ptracer主动调用detach分离，ptracer和tracee都没有出现task退出。

恢复原parent关系，唤醒stopped状态。

第二种状态就要复杂了。task退出会调用exit.c中的do_exit做退出清理。

使用ptrace_event通知ptracer，tracee要退出了。

如果当前为ptracer时forget_original_parent中退出ptraced的task调用detach还原parent，如果当前task的tracee同时task为线程组leader时通过ptracer。注意只有leader需要通知ptracer。当前task为EXIT_ZOMBIE状态，并不是EXIT_DEAD，需要等待wait收割EXIT_ZOMBIE状态。

tracee退出时do_notify_parent会通知ptracer，do_wait调用wait_task_zombie收割zombie。tracee状态EXIT_DEAD或EXIT_TRACE，仅当tracee为线程组leader时被设置为EXIT_TRACE，准备通知real_parent当前task退出。EXIT_DEAD后才是真正析构task。
